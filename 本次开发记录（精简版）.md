## 四、这次确认过的关键问题

### 1. 客户端启动崩溃

根因是 `TtsViewModel` 没正确接入 Hilt。

### 2. 服务端 422

根因是客户端没传 `character_name`。

### 3. 服务端 404

根因是角色虽然预加载了，但 API 真正读到的运行时状态不是同一份。

### 4. 客户端超时

根因是本地推理时间比默认 HTTP 超时更长。

### 5. 播放失败

根因是服务端实际返回了原始 PCM，而不是标准 WAV。

### 6. 连续点击后服务端不响应

根因是原来的“串行”只是全局锁，不是真正队列。

现在已经改成真实队列，问题明显改善。

### 7. 启动脚本容易因为位置不对或端口冲突直接失败

现在已经补了两层保护：

- 路径不对时直接提示正确使用方式
- 端口被占用时从 `8000` 开始自动递增直到可用

## 五、当前仍然存在的隐患

- TTS 服务协议仍然不够规范，客户端还在做兼容性兜底
- TTS 依赖局域网内电脑服务，IP/端口一变就要重新配置
- 角色名和参考音频路径仍然强依赖服务端配置
- 项目里还有一些旧的乱码文本没有清理
- 当前 Snackbar 调试提示更偏排障工具，不是最终产品形态
- 分发包继续瘦身的空间有限，不能再用大范围模式删除

## 六、关于分发包瘦身的结论

这次已经确认：

- 不能动多语言、模型转换、下载相关功能包
- 安全可删的主要只是少量运行环境冗余，例如：
  - `Doc`
  - `include`
  - `libs`
  - `pip`
  - `setuptools`
- 按目录名批量删 `test/docs/examples` 风险很高，容易误删运行所需资源

当前分发包保持在一个“可运行、不过度裁剪”的状态更稳。
