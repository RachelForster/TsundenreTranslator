## 三、这次排查出来的关键问题

### 1. 启动崩溃

最开始 App 不能进首页，是因为 `TtsViewModel` 没有正确接入 Hilt 注入。

### 2. 服务端参数不匹配

后面 TTS 请求报 `HTTP 422`，原因是服务端要求 `character_name`，但客户端没传。

### 3. 角色明明存在却报 404

之后报：

- `Character not found or reference audio not set.`

根因不是角色文件不存在，而是 `genie_tts` 的预加载角色状态和 API 服务实际读的状态不是同一份。  
这次已经通过修改本地启动脚本修掉。

### 4. 手机端超时

电脑在推理，但手机先超时了。  
原因是默认 HTTP 超时时间太短，这次已经加长。

### 5. 手机收到音频却播不出来

最后查明服务端虽然声明自己返回的是 `audio/wav`，但实际吐出来的是原始 PCM 数据。  
这就是 `MediaPlayer` 报错的根因。  
现在客户端已经做了兼容，会自动把原始 PCM 包装成 WAV 再播放。

## 四、当前项目仍然存在的隐患

下面这些问题还没有从根上解决，只是暂时绕过去了：

- 本地 TTS 服务的协议不严谨，声明和实际返回格式不一致
- TTS 强依赖局域网里的电脑服务，电脑 IP 变了就要改设置
- `TTS Character Name` 和 `TTS Ref Audio Path` 完全依赖服务端配置
- 代码里仍然有一些乱码注释和乱码字符串
- 现在的 Snackbar 调试提示是为了排障临时加的，不是最终产品体验
- `ChatViewModel` 里还有 unchecked cast 编译警告，后面最好清理
